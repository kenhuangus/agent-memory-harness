Why this exists

The problem

Long-running coding agents are stateless across sessions. They forget what they learned, so every session restarts cold — and that caps how well a smaller, cheaper model can ever do.

Memory is thrown away

Past fixes, codebase conventions, dead-ends and environment gotchas vanish at the end of a session. The same mistakes get repeated and the same context re-derived at cost.

Small models pay the most

A frontier model wins partly by holding more in its head at once. Without persistent memory, the only reliable way to get good long-horizon performance is to pay for the largest model.

💡

There is no standard, model-agnostic layer that decides what to remember, where to store it, how to retrieve it fast, and how to keep it consistent over time. Build that, and a cheaper model can borrow a bigger one's long-horizon advantage.

How we'll build it

Technical approach

A pluggable four-module memory harness over three indexed storage backends, wired into OpenCode and measured on existing public benchmarks — no home-grown evals.

💾

Persistence

Decides what/when/where to save; tags every item.

🧭

Router

Routes each query to the single best backend.

🔎

Orchestrator

Ranks by recency × relevancy, dedups, returns.

🌙

Zooming

Offline dedup, conflict resolution, governance.

Three storage backends, each indexed

  • Markdown + YAML — literal recall · inverted keyword index.
  • SQLite + vectors — semantic search · HNSW / FAISS ANN.
  • Graph store — relationships & conflicts · typed traversal.

Validated, not invented

Integrated into the open-source OpenCode agent. A controlled grid per benchmark runs Haiku + harness against Haiku, Opus 4.8 and Sonnet baselines — prompts and task order held fixed, every trajectory logged.

Boundaries

Scope

What the two-week sprint commits to — and, just as importantly, what it does not.

In scope

  • The memory harness: all four modules, end to end.
  • Three storage adapters behind one interface, each indexed.
  • The intelligent router (rule-based; learned-classifier hook).
  • The async zooming component: dedup, conflict resolution, governance, retention.
  • OpenCode integration for write/read on each agent step.
  • Eval harness on four benchmarks; baselines for Haiku, Opus 4.8, Sonnet.
  • Metric definitions, a reproducible protocol, and a results dashboard.

Out of scope (non-goals)

  • Building our own benchmark or dataset — we use public ones.
  • Training or fine-tuning any model.
  • Production hardening: multi-tenant infra, auth, SLAs, polished UI.
  • Distributed / scaled deployment — single-node is enough for the sprint.
  • Validating non-coding agents — design aims to generalize, not tested now.

Who owns what

Roles & deliverables

Person 1

🏗️ Harness Architecture & OpenCode Integration

Owns the critical path: the module contracts everyone else builds against.

Week 1
  • Design the three-module abstraction & storage interface (swappable backends).
  • Build the persistence layer: write logic, versioning, metadata tagging.
  • Instrument OpenCode to write memory on each agent step.
Week 2
  • Build the retrieval orchestrator (rank → dedup → return).
  • Wire in Person 3's router; run an end-to-end task.
Person 2

📊 Benchmarking & Evaluation

Owns experimental design, metrics, and reproducibility.

Week 1
  • Download & parse MemoryAgentBench, LongMemEval, SWE-ContextBench, SWE-Bench-CL.
  • Build data loaders + trajectory logging (what was written/retrieved/used).
  • Define the four metrics mathematically.
Week 2
  • Run baselines: Haiku no-memory, Opus 4.8 no-memory.
  • Statistical testing framework + results dashboard; document the protocol.
Person 3

🗄️ Storage Implementation & Router

Turns the abstraction into three real, fast backends + the dispatch layer.

Week 1
  • SQLite + vector pipeline (embedding model, HNSW/FAISS index).
  • Markdown store with YAML frontmatter + inverted index.
  • Graph store schema (Neo4j) with typed traversal index.
Week 2
  • Adapters for all three backends behind Person 1's interface.
  • Implement the router; performance-test each backend.
Person 4

🌙 Zooming Component & Memory Governance

Owns the offline engine that keeps memory clean and trustworthy.

Week 1
  • Deduplication logic (exact / semantic / near-duplicate).
  • Conflict detection + reconciliation rules (recency, confidence, source).
  • Session-filter & blacklist semantics.
Week 2
  • Async offline scheduler; must-know / must-do extraction.
  • Retention & pruning; observability logs; integration with P1 & P3.

Dependencies

⚠️

The hardest risk is semantic, not structural: what counts as "good memory." Person 2 and Person 4 must align on memory semantics in week 1, or the harness will store everything and retrieve nothing useful.

  • P1 → P3: the storage interface must be frozen by end of D3 so adapters can be built against it.
  • P3 → P1: the router lands before the orchestrator can route in week 2.
  • P2 → all: datasets + logging ready by D5 so baselines can start D6.
  • P1 + P3 → P4: the zooming worker integrates once real storage exists (D8+).
