Cookbook Memory

Persistent, self-curating memory that lets a small model think like a big one.

AI Agent Memory Harness · the bet: Haiku + memory closes the gap to Opus 4.8

The original idea

A cookbook for memory: what, when, where, how

Memory is leverage — but only if you keep the right things, at the right time, in the right place. That decision is the product.

How it works

Four modules, three backends, one loop

The agent writes as it works. A router sends each query to the one backend that answers best. An offline worker keeps the store clean.

When & how it's remembered

Daydream extracts. Dream consolidates.

A Stop hook fires the Daydream pass at session end; the deeper night Dream dedups, resolves contradictions, and sets governance across the whole store.

How we test improvement over time

Two benchmarks: one public, one our own

SWE-Bench-CL is a public continual-learning suite. VISTA is our own purpose-built memory-safety benchmark. Each is scored on its native metrics.

The 4-tier gated loop

Cheap gates first, expensive proof last

A full solve-rate run is hours and money. So every change is screened by two fast offline gates before it earns a Tier-3 measurement. Fail a gate → rejected, no spend.

The honest read

What's proven, what isn't

  • ✓
    Accumulated memory lifts solve rate when model & harness are fixed (SymPy +3/50; Django +1 per stage).
  • ✓
    Memory safety is solid — VISTA poisoning 1.0 · targeted-ASR 0.0 · RSI 1.0.
  • ~
    Dreaming isn't yet an outcome lift. It runs reliably and adds governance tags, but solve-rate after dreaming is unstable (Django 13→12; SymPy 48→45).
  • →
    The bottleneck moved. Memory is reached on nearly every task — retrieval quality & ranking, not availability, is now the limit.
  • !
    Segment before comparing. Cursor and Claude rows differ enough that mixing them misleads; isolate the run repo so benchmark agents can't touch the live tree.
