# ADR-harness-010: Commit the materialized release bundle + a root marketplace manifest so the plugin installs from git

**Status:** Accepted · **Date:** 2026-06-22 · **Contract:** no
**Supersedes:** ADR-harness-009 (one clause only — see below) · **Superseded by:** none

## Context
ADR-harness-009 established the right *authoring* model: one canonical `recall` skill
in the core, materialized into each harness bundle by a build step, delivered by a
single native install. It also declared the materialized bundle a **git-ignored build
output, never committed** — the bundle would be (re)built from source at
package/release time.

Productionizing the plugin — making `claude plugin install cookbook-memory` work for a
user who **does not have the repo** — exposed that the "never committed" clause is
incompatible with how Claude Code installs a plugin from git. Verified against the
Claude Code plugin system (2026-06-22):

1. `claude plugin marketplace add <git-url>` reads **`.claude-plugin/marketplace.json`
   at the repo ROOT only** — you cannot point `add` at a subdirectory.
2. A plugin's skill files **must physically exist inside the plugin's `source`
   directory** at install time. `claude plugin install` **copies** the plugin source
   into `~/.claude/plugins/cache/`; paths referencing files *outside* the plugin
   directory do not survive the copy. So the skill cannot be referenced from elsewhere
   in the repo at install — it must be sitting in the bundle.

Together these mean: for a repo-less install, the **materialized bundle (manifests +
MCP + hooks + the copied skill) must be present in the checked-out git tree** at a
stable path, and a **root marketplace manifest** must point at it. A git-ignored
`dist/` that only exists after a local `build-bundle` run satisfies neither — a user
who only has the git URL gets no marketplace and no skill.

This does **not** disturb ADR-009's authoring model: the skill is still authored
exactly once in the core, the bundle is still *generated* by the build step (not
hand-edited), and the user still runs one native install. The single clause that must
change is "git-ignored, never committed."

## Options considered
- **Commit the materialized bundle + a root marketplace manifest** (chosen): the build
  step writes the bundle to a *tracked* path (`plugin/marketplace/cookbook-memory/`),
  and that path is committed — a generated-but-checked-in artifact, like a lockfile or
  a vendored client. A root `.claude-plugin/marketplace.json` points at it via a
  `git-subdir` source. `claude plugin marketplace add <repo>` then works with no clone,
  no local build. The bundle is regenerated by the build step and a CI check fails if
  the committed copy drifts from the canonical skill.
- **CI builds and commits the bundle to a release branch/tag**: honors ADR-009's
  "never hand-commit" spirit most literally — source stays bundle-free; a release job
  materializes and pushes to a `release` ref. Rejected for now: it needs release-CI
  wiring that does not yet exist, and `marketplace add <repo>@release` is a less
  obvious install URL than the default branch. Revisit if hand-commit drift becomes a
  real problem.
- **Separate plugin repo**: host the bundle in its own GitHub repo. Cleanest install
  URL, but a second repo to publish and keep in sync, outside this team repo's
  conventions and CODEOWNERS. Rejected as premature.

## Decision
**Commit the materialized release bundle to a tracked path and add a repo-root
marketplace manifest pointing at it**, so the plugin installs from git with no clone:

- The build step (`memory-cli build-bundle`) materializes the bundle to
  **`plugin/marketplace/cookbook-memory/`** — manifests + `.mcp.json` + `hooks/` +
  the copied `skills/recall/SKILL.md`. **This path is committed** (removed from
  `.gitignore`'s `dist/` rule by living outside `dist/`).
- A **`.claude-plugin/marketplace.json` at the repo root** declares the
  `cookbook-memory` plugin with a `git-subdir` source pointing at that path.
- The user installs with:
  `claude plugin marketplace add https://github.com/kenhuangus/agent-memory-harness`
  then `claude plugin install cookbook-memory` — no repo checkout, no local build.
- The committed bundle is a **generated artifact**: it is produced by the build step
  from the one canonical skill, never hand-edited, and a CI/test check asserts it
  matches a fresh `build-bundle` (so a stale skill cannot ship silently).

This **supersedes only the "git-ignored, never committed" clause of ADR-009**.
Everything else in ADR-009 stands: one canonical authored skill, build-time
materialization, single native install, harness-agnostic skill content.

## Rationale
A repo-less install is the whole point of shipping a plugin; the Claude Code plugin
system's two hard constraints (root manifest, skill co-located inside the copied
source) leave no way to install from git without a committed bundle. Committing a
*generated* artifact is a well-understood pattern (lockfiles, vendored SDKs,
`dist/`-in-git for libraries that ship built output) — the duplication ADR-009 feared
was *authoring* duplication (N hand-maintained skill copies), which this does not
reintroduce: there is still exactly one authored skill, and the committed copy is
mechanically derived and drift-checked. Keeping the marketplace on the existing team
repo (vs. a separate repo or a release branch) gives the simplest possible install URL
today with the least new infrastructure.

## Tradeoffs & risks
- **Drift**: a committed generated artifact can fall out of sync with its source if
  someone edits the skill and forgets to rebuild. Mitigation: a test/CI check
  rebuilds the bundle and asserts byte-equality with the committed copy, failing the
  PR on drift — the same guard ADR-009 already implies for the build step, now
  pointed at a tracked path instead of a clean check.
- **A generated file in git review noise**: bundle changes show up in diffs. Accepted —
  it is small (manifests + one SKILL.md) and changes only when the skill or manifests
  change, which are reviewable events anyway.
- **Two skill bytes in the tree** (canonical core copy + committed bundle copy): this
  is the narrow, deliberate cost of git-install. It is *generated* duplication, not
  authored, and drift-guarded — distinct from the hand-maintained N-copy duplication
  ADR-009 forbids.
- **`git-subdir` source support is version-dependent** in Claude Code; if an older
  client can't resolve it, the fallback is `marketplace add` against a clone. Noted in
  the README install section.

## Consequences for the build
- **Policy:** the release bundle is materialized to the **tracked** path
  `plugin/marketplace/cookbook-memory/` and **committed**. The old `dist/`-based,
  git-ignored bundle remains valid for ad-hoc local builds (`build-bundle --out dist/…`),
  but the *shipping* bundle is the committed one.
- **Policy:** a repo-root `.claude-plugin/marketplace.json` is the source of truth for
  `marketplace add <repo>`; it lists `cookbook-memory` via a `git-subdir` source at the
  committed bundle path.
- **Policy:** a test asserts the committed bundle equals a fresh `build-bundle` output
  (drift guard), so the canonical skill remains the single source of truth in practice,
  not just on paper.
- **Unchanged from ADR-009:** one authored skill in the core; build-time
  materialization; single native install; skill content is harness-agnostic.
